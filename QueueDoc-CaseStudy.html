System Architecture
QueueDoc
04 / 08
Built for Real-Time Reliability
Every architectural decision in QueueDoc was made to maximize data integrity, minimize latency, and handle failure cases gracefully. The system is stateless on the client and authoritative on the server.
Layer 01
Frontend — React + Vite

Two independent React SPAs: Reception Dashboard and Patient Display. Both mount a shared Socket.io client and listen for server-pushed state. No polling. No client-initiated fetches post-mount.

↓
Layer 02
Socket.io Engine

Handles all bidirectional communication. Clients emit action events. Server emits state. The engine manages reconnection, room broadcasting, and event debouncing per socket.

↓
Layer 03
Express Route Handlers

Processes all socket events. Applies business logic (priority sorting, EMA calculation, audit logging). Every handler calls broadcastUpdate() after completing a mutation.

Layer 04
MongoDB — Document Store

Four collections: tokens, counters, clinicconfigs, auditlogs. The counter collection uses native $inc for atomic sequence generation. All writes are confirmed before broadcast.

↓
Layer 05
EMA Wait-Time Module

A standalone calculation module fed real consultation durations. Applies Exponential Moving Average (α=0.3) with outlier rejection at 3× median. Result included in every broadcast.

↓
Layer 06
Audit Logger

Every queue mutation is persisted to AuditLog collection with event_type, token reference, description, and timestamp. Streamed live to the receptionist dashboard via audit:update socket event.

Core Features
QueueDoc
05 / 08
Designed for Real Clinic Operations
Every feature in QueueDoc was designed to solve a documented clinic failure mode — not an imagined one. The following nine capabilities cover the complete surface of real outpatient queue management.
Core

Atomic Token Generation

MongoDB $inc on a dedicated Counter document makes duplicate tokens structurally impossible — even under simultaneous concurrent submissions.

Core

Real-Time Queue Sync

Every mutation triggers a full-state broadcast to all connected clients. Sub-100ms propagation. No polling. All screens display identical data from one source of truth.

Priority

Urgent Patient Override

Patients registered as Urgent are automatically sorted above all standard patients. Queue re-orders on every broadcast without manual repositioning.

UX

Smart Hold / Return

Patients who step out are placed in "holding" state. When the queue shrinks to 3 ahead, the server auto-reinserts them — preserving turn order without manual intervention.

Communication

Doctor Delay Broadcast

One button adds a 15-minute offset to ClinicConfig, broadcasts a delay banner to all patient displays, and updates all wait-time estimates simultaneously.

Automation

Auto No-Show Timeout

A server-side timer fires 2 minutes after every Call Next. Unresponsive patients are auto-transitioned to no_show, logged, and the queue advances without receptionist action.

Intelligence

EMA Wait-Time Algorithm

Tracks real consultation durations. Applies Exponential Moving Average (α=0.3) with outlier rejection at 3× median. Self-corrects as doctor's pace evolves throughout the day.

Accessibility

Voice Announcements

Browser-native Web Speech API announces each called token out loud. Zero backend cost. Ensures patients are notified even when not watching the display.

Observability

Live Audit Trail

Every queue action persisted to MongoDB AuditLog. Streamed live to the receptionist with time-diff badges — verifiable proof of system speed and queue integrity.

Concurrency, Edge Cases
& Measurable Outcomes
QueueDoc was built to handle the failure modes, not just the happy path. Below are the specific concurrency scenarios addressed, and the measurable outcomes delivered.
Concurrency Guarantees
Duplicate Token Assignment
MongoDB atomic $inc on Counter collection. Database engine serializes operations natively.
Concurrent Call-Next Clicks
findOneAndUpdate({ status:'waiting' }) — DB guard ensures exactly one request succeeds. Second request receives null and exits cleanly.
Browser Crash / Reconnect
Every Socket.io 'connection' event triggers full queue:full_state broadcast. UI fully rebuilt from DB state.
No-Show Patient Timeout
2-minute server-side setTimeout per call. DB re-check before auto-transition prevents false positives.
Acknowledged Limitation: The 600ms in-memory debounce Set does not replicate across horizontally-scaled Node.js instances. The DB-level atomic lock handles data integrity at any scale. In a production multi-node environment, the debounce would be replaced with a Redis-backed distributed lock.

Wait-Time Intelligence
Static estimates destroy patient trust. QueueDoc uses an Exponential Moving Average algorithm that adapts to the doctor's actual pace in real-time.
EMA Formula
EMA(t) = α × duration(t) + (1 − α) × EMA(t−1)
where α = 0.3 — 30% weight on latest consultation, 70% on running average. Self-corrects as pace changes.

Future Roadmap
QueueDoc
08 / 08
What's Next
QueueDoc has validated the core real-time infrastructure. The following phases expand the product from a single-clinic tool to a scalable SaaS platform serving India's entire outpatient healthcare segment.
Phase 1
SMS & WhatsApp Notifications

Notify patients on their phones when they are 2nd in line. Reduce in-clinic crowding and increase walk-away-and-return reliability. Integration via Twilio or WhatsApp Business API.

Phase 2
Multi-Doctor Load Balancing

Auto-assign incoming tokens to the doctor queue with the shortest current wait. Extends the architecture to multi-queue scheduling — the single most-requested clinic feature.

Phase 3
Analytics Dashboard

Historical reporting on patient volume by hour, average consultation duration by day, and no-show rate trends. Enables clinic operators to make data-driven scheduling decisions.

Phase 4
AI-Based Wait Forecasting

Train a time-series model on accumulated consultation data to predict queue wait time within a 2-minute accuracy band by time of day, doctor, and patient type.

Phase 5
Cloud SaaS Platform

Multi-tenant deployment on Railway / Render with MongoDB Atlas. Onboard any clinic in under 5 minutes. Subscription model targeting the 5.65 lakh registered outpatient facilities across India.
